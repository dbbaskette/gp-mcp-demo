1. High-Level Overview

gp-mcp-server (MCP) aims to expose statistical and machine learning capabilities in Greenplum via a unified MCP (Model Context Protocol) interface.

These capabilities come from three distinct components , each with different responsibilities:

  • Apache MADlib — in-database statistical and ML algorithms
  • PostgresML — SQL-accessible Python-based ML
  • gpMLBot — AutoML workflow orchestrator built on top of MADlib and PostgresML

MCP should unify access without re-implementing ML logic.

2. Component Responsibilities and Examples

2.1 Apache MADlib — In-Database Analytics and ML Library

What it is:

  • In-database analytics and ML library implemented as SQL functions / UDFs
  • Optimized for large datasets and parallel execution

Strengths / Typical Use Cases:

  • Descriptive statistics (mean, median, stddev, percentiles)
  • Correlation and covariance
  • Linear models, logistic regression, K-means clustering

Example SQL:


SELECT * FROM madlib.summary(
  'public.sales_data',
  'revenue'
);


MCP Principle: MCP can call MADlib directly for statistical analysis or single-model ML tasks.

2.2 PostgresML — SQL Interface to Python ML Libraries

What it is:

  • Database extension allowing SQL to invoke Python ML libraries (scikit-learn, XGBoost, LightGBM)
  • Provides ML capabilities that MADlib doesn’t support

Strengths / Typical Use Cases:

  • Tree-based models (XGBoost, Random Forests)
  • Algorithms not available in MADlib

MCP Principle: MCP does not call PostgresML directly ; PostgresML is called by gpMLBot for AutoML workflows.

Example SQL (via gpMLBot):

SELECT pgml.train(
  project_name => 'churn_model',
  algorithm => 'xgboost',
  relation_name => 'customer_features'
);

2.3 gpMLBot — Automated Machine Learning Orchestrator

What it is:

  • AutoML workflow engine that orchestrates MADlib and PostgresML
  • Automates data splitting, algorithm selection, hyperparameter tuning, and model ranking

Strengths / Typical Use Cases:

  • End-to-end ML workflow automation
  • Leaderboards and experiment tracking
  • Guided CLI / conversational interface

Example Bash CLI:

gpmlbot train
  --config config.toml
  --non-interactive

MCP Principle: MCP calls gpMLBot to run AutoML workflows; gpMLBot chooses MADlib or PostgresML as needed.

3. Relationship Between Components

  • MCP (top layer) : unified API/tool interface
  • gpMLBot (middle layer) : orchestrates AutoML, calls MADlib and PostgresML
  • MADlib & PostgresML (execution layer) : run actual algorithms
  • Greenplum Database (bottom layer) : data storage and parallel computation

Key Principle:

  • MADlib and PostgresML are algorithm execution engines
  • gpMLBot is the AutoML orchestrator
  • MCP is the user-facing entry point

4. Best Practices for MCP Integration

  1. Thin orchestration layer: generate SQL/CLI commands, execute, normalize results
  2. Explicit capabilities: one MCP tool per backend functionality
  3. Treat gpMLBot as the source of truth for AutoML state (model IDs, experiments, leaderboards)
  4. Declarative inputs: describe what to do, not how to do it
  5. Direct MADlib calls allowed for statistics or single-model ML
  6. Do not implement ML algorithms in MCP

5. Prerequisites

Before using gp-mcp-server's AutoML capabilities (gpMLBot), SSH configuration is required. SSH is used to execute the gpmlbot CLI command on the Greenplum coordinator host. Note: SSH is only required for AutoML features (gpmlbot_train, gpmlbot_predict, gpmlbot_list_models). MADlib statistical tools work without SSH.

7. Example MCP Tool/Prompt Definitions

7.1 Analytical Statistics tool (MADlib)

Tool: get_table_madlib_analytics
Purpose: descriptive statistics & data quality
Inputs: database, schema, table, columns and etc
Outputs: count, min/max, mean, median, stddev, percentiles, distinct count, null fraction

Example JSON:

{
  "tool": "get_table_madlib_analytics",
  "arguments": {
    "database": "analytics",
    "schema": "public",
    "table": "sales",
    "columns": "revenue, discount",
    "include_correlation": true,
    "include_covariance": true,
    "include_quartiles": true,
    "top_n_values": 20
  }
}

7.2 AutoML Training Tool (gpMLBot)

Tool: gpmlbot_train
Purpose: Run AutoML training workflow — trains one or more algorithms via the gpmlbot CLI on the Greenplum coordinator host (over SSH).
Required inputs: database, schema, table, target_column, feature_columns (array), task_type (classification or regression)
Optional inputs: id_column (default: id), algorithms (array; defaults to all MADlib algorithms for the task type), algorithm_params (per-algorithm hyperparameter map), test_size (default: 0.2)
Outputs: Per-algorithm training results with leaderboard rows added to gpmlbot.models. Use gpmlbot_list_models to retrieve the best model ID.
Note: pgml algorithms (xgboost, random_forest, lightgbm, etc.) are not supported on GP6. When omitted, only MADlib algorithms are used by default.

Example JSON:

{
  "tool": "gpmlbot_train",
  "arguments": {
    "database": "analytics",
    "schema": "public",
    "table": "customer_churn",
    "id_column": "customer_id",
    "feature_columns": ["age", "tenure", "monthly_spend"],
    "target_column": "churn",
    "task_type": "classification",
    "algorithms": ["decision_tree", "multilayer_perceptron", "support_vector_machines"]
  }
}

Async execution: Training can take minutes to hours. The MCP server monitors execution for 5 seconds: if it completes within that window the result is returned immediately; otherwise it returns a "Training started in background" message and sends a notifications/gpmlbot_result MCP notification when all algorithms finish.

7.3 Prediction Tool (gpMLBot)

Tool: gpmlbot_predict
Purpose: Apply a trained model to new data, writing predictions to an output table.
Required inputs: model_id (number), database, input_schema, input_table
Optional inputs: id_column (auto-detected if omitted), output_schema (defaults to input_schema), output_table (auto-generated as <input_table>_predictions_<timestamp> if omitted)
Outputs: Prediction results written to output_schema.output_table; returns a summary of rows processed and prediction statistics.

Example JSON:

{
  "tool": "gpmlbot_predict",
  "arguments": {
    "model_id": 1,
    "database": "analytics",
    "input_schema": "public",
    "input_table": "new_customers",
    "id_column": "customer_id",
    "output_schema": "public",
    "output_table": "new_customers_predictions"
  }
}

Async execution: Like training, prediction also uses the 5-second monitor pattern and sends a notifications/gpmlbot_result notification when complete if it takes longer.

7.4 AutoML Model Retrieval (gpMLBot)

Tool: gpmlbot_list_models
Purpose: Retrieve results and metadata for trained models, optionally filtered by source table.
Inputs: schema (optional), table (optional), limit (default: 1000), offset (for pagination)
Outputs: Paginated models list with fields: id, date_created, trained_table_schema, trained_table_name, algorithm_type, extension, algorithm, accuracy, precision, madlib_output_table, madlib_summary_table, madlib_additional_tables, pgml_label, hyperparams

Example JSON:


{
  "tool": "gpmlbot_list_models",
  "arguments": {
    "schema": "public",
    "table": "customer_churn"
  }
}

​​8.  Handling Missing Components (MADlib / gpMLBot / PostgresML)

In real-world deployments, it is possible that MADlib, gpMLBot, or PostgresML may not be installed on some Greenplum instances. To ensure robustness of MCP tools, the following strategies are recommended:

  1. Component Detection
    • MADlib:
      SELECT * FROM pg_extension WHERE extname='madlib';
    • gpMLBot:
      SELECT EXISTS (SELECT 1 FROM information_schema.tables WHERE table_schema = 'gpmlbot' AND table_name = 'models');
    • PostgresML:
      SELECT * FROM pg_extension WHERE extname='pgml';
    • Before executing any MCP tool, verify whether the required component is available.
    • Example checks:
  2. Conditional Execution
    • If MADlib is missing, MCP tools relying on in-database statistics or single-model ML should be disabled or return a clear error.
    • If gpMLBot is missing, AutoML workflows should display a warning or skip execution.
    • PostgresML is only called via gpMLBot; if missing, gpMLBot can fallback to MADlib algorithms.
    • Adjust MCP tool behavior based on availability:
  3. User Feedback and Logging
    • Provide clear messages or logs to inform users of missing components.
    • Example:
      "MADlib extension is not installed. Analytical statistics tools are disabled."
  4. Optional Fallback / Degraded Functionality
    • Descriptive statistics can use standard SQL aggregate functions if MADlib is unavailable.
    • Simple regression models can use MADlib as a replacement for PostgresML models.
    • Offer partial functionality where feasible.

In summary: MCP tools should check dependencies upfront , decide the appropriate execution path based on availability, and provide clear warnings or fallback options to maintain usability and robustness.

9. Summary

  • MADlib: scalable in-database analytics and ML, callable directly by MCP
  • PostgresML: Python ML models, only called by gpMLBot
  • gpMLBot: AutoML orchestrator, calls MADlib + PostgresML
  • gp-mcp-server: top-level integration layer exposing MCP tools

Clear boundaries reduce maintenance and risk and prevent redundant implementation.
