Why OAuth 2.1 for GPDB MCP Server?

The Limitations of Basic Authentication

Traditional username/password authentication creates significant friction and risk in enterprise environments:

  • Elevated Security Risk: Storing and transmitting credentials increases the attack surface. Basic Auth lacks modern safeguards like PKCE, short-lived tokens, and instant revocation, leaving the system vulnerable to brute-force attacks and credential theft.
  • Identity Fragmentation: Without Single Sign-On (SSO) support, the MCP server becomes an "identity island." It cannot leverage existing providers like Active Directory or Okta, leading to disconnected user lifecycles and manual provisioning.
  • Operational Bottlenecks: Manual password resets, rotation policies, and RBAC mapping do not scale. These labor-intensive processes are error-prone and fail to reflect real-time organizational changes.
  • Compliance Deficiencies: Basic Auth offers minimal visibility into authentication events. This lack of centralized logging makes it difficult to meet rigorous audit standards such as SOC 2, GDPR, or HIPAA. 

The OAuth 2.1 Solutions

OAuth 2.1 modernizes the MCP server by replacing legacy credentials with a standards-based framework:

  • Hardened Security: By utilizing cryptographically signed, short-lived tokens and PKCE, the system eliminates password exposure. Centralized identity management enables instant, system-wide session revocation.
  • Deep Enterprise Synergy: Seamlessly integrates with existing infrastructure (Okta, Keycloak, Auth0) via SSO. This establishes a "single source of truth," ensuring user permissions and identities stay synchronized automatically.
  • Automated Scalability: Features like Dynamic Client Registration and claim-based role mapping remove the IT burden of manual user creation, allowing the MCP server to scale effortlessly with the organization.
  • Frictionless Experience: Users authenticate through their familiar corporate login portals. SSO eliminates password fatigue and provides a consistent, mobile-friendly access flow across all services.

Introduction to OAuth 2.1

OAuth 2.1 is an authorization framework that enables applications to obtain limited access to user accounts on an HTTP service. It provides a secure and standardized way for users to grant third-party applications access to their resources without sharing their passwords.

Key Concepts

  • Authorization Server: The server that issues access tokens after successfully authenticating the user and obtaining authorization. Examples include Auth0, Keycloak, WorkOS, and Okta.
  • Resource Server: The server hosting protected resources (in our case, the GPDB MCP Server) that accepts and responds to protected resource requests using access tokens.
  • Client: The application requesting access to a protected resource on behalf of the resource owner (user).
  • Access Token: A credential that can be used by an application to access an API. It represents the authorization granted to the client by the resource owner.
  • Bearer Token: A type of access token that is included in HTTP requests using the Authorization: Bearer <token> header.

OAuth 2.1 Architecture Overview 

The GPDB MCP Server implements OAuth 2.1 in delegated mode, where the MCP server acts as a Resource Server only. All OAuth flows are delegated to an external Authorization Server (e.g., Keycloak, Auth0, WorkOS).

How It Works: Technical Overview

When the MCP server receives a request with a Bearer token, it performs the following validation:

  1. Extract Bearer Token: Parses the Authorization: Bearer <token> header
  2. JWKS Signature Verification:
    • Fetches public keys from the authorization server's JWKS endpoint
    • Verifies the token signature using the appropriate public key
    • Caches JWKS for performance (automatic refresh)
  3. Claim Validation:
    • Issuer (iss): Must match the authorization server URL
    • Audience (aud): Must match the configured resource URL
    • Expiration (exp): Token must not be expired
    • Subject (sub): User identifier
  4. Permission Mapping: Maps OAuth2 roles/groups to database credentials
  5. Database Connection: Establishes connection to GPDB using mapped credentials

Delegated Mode Architecture

The GPDB MCP Server operates in delegated mode, meaning:

  • No User Authentication: The MCP server does not authenticate users directly
  • No Token Issuance: The MCP server does not issue access tokens
  • Token Validation Only: The MCP server validates tokens issued by external authorization servers
  • Resource Server Role: The MCP server acts as a resource server that protects MCP endpoints

This architecture provides:

  • Separation of Concerns: Authentication logic is handled by dedicated identity providers
  • Enterprise Integration: Works with existing SSO solutions
  • Scalability: Authorization servers can handle high authentication loads
  • Compliance: Meets enterprise security and audit requirements

User Configuration Guide

Prerequisites

Before configuring OAuth 2.1, ensure you have:

  1. Authorization Server: Access to an OAuth 2.1-compliant authorization server (Keycloak, Auth0, WorkOS, etc.)
  2. Client Registration: Either a pre-registered OAuth client or support for dynamic client registration
  3. Database Users: Database users in GPDB that correspond to your permission levels
  4. TLS Certificates (optional): For mTLS configuration

Configuration File Structure

The OAuth 2.1 configuration is defined in your MCP server configuration file (YAML):

mcp:
  host: "127.0.0.1"
  port: 3000
  endpoint: "/mcp"
  tls_enable: true
  tls_cert: "/path/to/server.crt"
  tls_key: "/path/to/server.key"

service:
  gpdb:
    host: "localhost"
    port: 5432
    database: "postgres"
    username: "gpadmin"  # Service account (not used for OAuth2 users)
    password: ""

auth:
  enabled: true
  type: "oauth2"
  rate_limit_failures: 10
  oauth2:
    resource_url: "https://127.0.0.1:3000/mcp"
    scopes_supported:
      - "mcp:tools"
      - "mcp:resources"
    authorization_server_url: "https://keycloak.example.com/realms/master"
    jwks_url: "https://keycloak.example.com/realms/master/protocol/openid-connect/certs"
    audience: "https://127.0.0.1:3000/mcp"
    permission_levels:
      - name: "readonly"
        roles:
          - "readonly"
        database_username: "readonly_user"
        database_password: ""
    default_database_username: "readonly_user"
    default_database_password: ""

Step-by-Step Configuration

Step 1: Configure MCP Server Basic Settings

mcp:
  host: "127.0.0.1"          # Server bind address
  port: 3000                  # Server port
  endpoint: "/mcp"            # MCP endpoint path
  tls_enable: true            # Enable TLS (recommended for production)
  tls_cert: "/path/to/server.crt"
  tls_key: "/path/to/server.key"

Important Notes:

  • If tls_enable: true, use https:// in resource_url and audience
  • If tls_enable: false, use http:// (only for localhost development)

Step 2: Configure Authorization Server

auth:
  oauth2:
    authorization_server_url: "https://keycloak.example.com/realms/master"
    jwks_url: "https://keycloak.example.com/realms/master/protocol/openid-connect/certs"

Authorization Server URLs by Provider:

  • Keycloak: https://keycloak.example.com/realms/<realm-name>
  • Auth0: https://<tenant>.auth0.com
  • WorkOS: https://api.workos.com
  • Okta: https://<tenant>.okta.com/oauth2/<authorization-server-id>

JWKS URL Auto-Detection:

  • If jwks_url is not specified, the server will attempt to auto-detect it:
    • Keycloak: {authorization_server_url}/protocol/openid-connect/certs
    • Auth0: {authorization_server_url}/.well-known/jwks.json
    • Standard OAuth2: {authorization_server_url}/.well-known/jwks.json

Step 3: Configure Resource URL and Audience

auth:
  oauth2:
    resource_url: "https://127.0.0.1:3000/mcp"
    audience: "https://127.0.0.1:3000/mcp"

Important:

  • resource_url: The full URL clients use to connect to the MCP server
  • audience: Must match the aud claim in access tokens
  • Both must use the same protocol (http/https) and match the actual server URL
  • For production, use your actual domain: https://mcp.example.com/mcp

Step 4: Configure Scopes

auth:
  oauth2:
    scopes_supported:
      - "mcp:tools"
      - "mcp:resources"

Standard Scopes:

  • mcp:tools: Access to MCP tools (database queries, etc.)
  • mcp:resources: Access to MCP resources

You can define custom scopes based on your authorization server's configuration.

Step 5: Configure Permission Levels

See Permission-Based User Mapping section for detailed configuration.

Step 6: Configure Default Database User

auth:
  oauth2:
    default_database_username: "readonly_user"
    default_database_password: ""

This user is used when:

  • No permission level matches the user's roles/groups
  • Permission mapping is not configured

Security Note: Use a user with minimal privileges as the default.

Environment Variables

You can also configure OAuth 2.1 using environment variables:

export AUTH_ENABLED=true
export AUTH_TYPE=oauth2
export AUTH_OAUTH2_RESOURCE_URL=https://127.0.0.1:3000/mcp

export AUTH_OAUTH2_DEFAULT_DATABASE_USERNAME=readonly_user

Common Configuration Issues

Issue 1: Token Validation Fails

Symptoms: 401 Unauthorized even with valid token

Possible Causes:

  • audience mismatch: Token aud claim doesn't match configured audience
  • issuer mismatch: Token iss claim doesn't match authorization_server_url
  • Token expired: Check token expiration time
  • JWKS URL incorrect: Verify jwks_url is accessible

Solution:

# Verify these match your token claims
audience: "https://127.0.0.1:3000/mcp"  # Must match token 'aud' claim
authorization_server_url: "https://keycloak.example.com/realms/master"  # Must match token 'iss' claim

Issue 2: Permission Mapping Not Working

Symptoms: All users use default database user

Possible Causes:

  • Permission levels not configured
  • Role/group names don't match token claims
  • Token doesn't contain expected claims

Solution:

  • Enable debug logging: log-level: "debug"
  • Check token claims: Decode JWT token and verify roles and groups claims
  • Verify permission level configuration matches token structure

Issue 3: TLS Certificate Errors

Symptoms: x509: certificate signed by unknown authority

Solution: Configure TLS for authorization server connection (see Mutual TLS Configuration)

Permission-Based User Mapping

Permission-Based User Mapping allows the MCP server to map OAuth2 user identities (roles and groups) to specific database users. This enables fine-grained access control where different OAuth2 users connect to GPDB using different database credentials based on their permissions.

How It Works

The matching algorithm works as follows:

  1. Extract Claims: Extract roles and groups from the JWT token
  2. Iterate Levels: Check each permission level in order
  3. Role Matching: If a permission level specifies roles, check if any user role matches
  4. Group Matching: If a permission level specifies groups, check if any user group matches
  5. Match Criteria:
    • If both roles and groups are specified: Match if either role OR group matches (OR logic)
    • If only roles is specified: Match if any role matches
    • If only groups is specified: Match if any group matches
  6. First Match Wins: The first matching level is used (order matters!)

Configuration Example

auth:
  oauth2:
    permission_levels:
      # Level 1: Readonly users (checked first)
      - name: "readonly"
        roles:
          - "readonly"
          - "viewer"
        groups:
          - "readonly-group"
        database_username: "readonly_user"
        database_password: ""
        
      # Level 2: Analyst users
      - name: "analyst"
        roles:
          - "analyst"
          - "data-analyst"
        database_username: "analyst_user"
        database_password: ""
        
      # Level 3: Developer users
      - name: "developer"
        roles:
          - "developer"
          - "dev"
        groups:
          - "engineering"
        database_username: "dev_user"
        database_password: ""
        
      # Level 4: Admin users (checked last)
      - name: "admin"
        roles:
          - "admin"
          - "dba"
          - "administrator"
        database_username: "admin_user"
        database_password: ""
    
    # Default user if no permission level matches
    default_database_username: "readonly_user"
    default_database_password: ""

Step-by-Step Configuration

Step 1: Create Database Users in GPDB

Create database users that correspond to your permission levels:

-- Readonly user
CREATE USER readonly_user WITH PASSWORD 'secure_password';
GRANT CONNECT ON DATABASE postgres TO readonly_user;
GRANT USAGE ON SCHEMA public TO readonly_user;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly_user;

-- Analyst user
CREATE USER analyst_user WITH PASSWORD 'secure_password';
GRANT CONNECT ON DATABASE postgres TO analyst_user;
GRANT USAGE ON SCHEMA public TO analyst_user;
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO analyst_user;

-- Developer user
CREATE USER dev_user WITH PASSWORD 'secure_password';
GRANT CONNECT ON DATABASE postgres TO dev_user;
GRANT ALL PRIVILEGES ON SCHEMA public TO dev_user;

-- Admin user
CREATE USER admin_user WITH PASSWORD 'secure_password';
GRANT ALL PRIVILEGES ON DATABASE postgres TO admin_user;

Step 2: Configure Roles/Groups in Authorization Server

Configure roles and groups in your authorization server (Keycloak, Auth0, etc.):

Keycloak Example:

  1. Go to Realm → Roles → Create Role: readonly, analyst, developer, admin
  2. Go to Groups → Create Group: readonly-group, engineering
  3. Assign roles to users or groups

Step 3: Configure Permission Levels in MCP Server

auth:
  oauth2:
    permission_levels:
      - name: "readonly"
        roles:
          - "readonly"
        database_username: "readonly_user"
        database_password: "secure_password"
        
      - name: "analyst"
        roles:
          - "analyst"
        database_username: "analyst_user"
        database_password: "secure_password"
        
      - name: "developer"
        roles:
          - "developer"
        database_username: "dev_user"
        database_password: "secure_password"
        
      - name: "admin"
        roles:
          - "admin"
        database_username: "admin_user"
        database_password: "secure_password"
    
    default_database_username: "readonly_user"
    default_database_password: "secure_password"

Step 4: Test Permission Mapping

  • Get Access Token with user that has readonly role
  • Connect to MCP Server

  --insecure \
  -H "Authorization: Bearer <token-with-readonly-role>"
  • Check Server Logs: Verify that readonly_user is used for database connection
  • Test Different Roles: Repeat with users having different roles

Advanced Configuration

Multiple Roles/Groups per Level

You can specify multiple roles or groups for a single permission level:

permission_levels:
  - name: "readonly"
    roles:
      - "readonly"
      - "viewer"
      - "guest"
    groups:
      - "readonly-group"
      - "public"
    database_username: "readonly_user"

Matching: User matches if they have any of the specified roles OR any of the specified groups.

Role-Only or Group-Only Matching

You can configure levels that match only on roles or only on groups:

permission_levels:
  # Role-only matching
  - name: "analyst"
    roles:
      - "analyst"
    # No groups specified - only matches on roles
    database_username: "analyst_user"
    
  # Group-only matching
  - name: "engineering"
    groups:
      - "engineering"
      - "dev-team"
    # No roles specified - only matches on groups
    database_username: "dev_user"

Order Matters

Permission levels are evaluated in order. The first matching level is used:

permission_levels:
  # This will match users with "admin" role first
  - name: "admin"
    roles:
      - "admin"
    database_username: "admin_user"
    
  # This will never match users with "admin" role
  # (because admin level matches first)
  - name: "readonly"
    roles:
      - "readonly"
      - "admin"  # This won't work for admin users!
    database_username: "readonly_user"

Best Practice: Order permission levels from most specific to least specific.

Troubleshooting

Issue1: All Users Use Default Database User

Check:

  1. Permission levels are configured correctly
  2. Token contains expected roles or groups claims
  3. Role/group names match exactly (case-sensitive)
  4. Enable debug logging: log-level: "debug"

Debug Steps:

# Decode JWT token to see claims
echo "<token>" | cut -d. -f2 | base64 -d | jq

# Check server logs for permission mapping
tail -f ~/.gp_mcp/server.log | grep -i "permission"

Issue2: Wrong Database User Selected

Check:

  1. Permission level order (first match wins)
  2. Multiple roles/groups in token (may match unexpected level)
  3. OR logic behavior (role OR group match)

Solution: Reorder permission levels or make matching more specific.

Mutual TLS (mTLS) Configuration

Mutual TLS (mTLS) provides an additional layer of security by requiring both the client and server to present certificates during the TLS handshake. For OAuth 2.1, mTLS can be used in two scenarios:

  1. MCP Server to Authorization Server: The MCP server presents a client certificate when connecting to the authorization server (for JWKS fetching, token validation)
  2. MCP Server TLS: The MCP server itself can require client certificates from MCP clients (separate from OAuth2)

This section focuses on mTLS for MCP Server to Authorization Server connections.

When to Use mTLS

Use mTLS when:

  • Private Authorization Server: Your authorization server is behind a firewall and requires client certificates
  • Enhanced Security: You want to ensure only authorized MCP servers can fetch JWKS or validate tokens
  • Compliance Requirements: Your organization requires mTLS for service-to-service communication
  • Self-Signed Certificates: Your authorization server uses self-signed certificates and you want mutual authentication

Configuration

Step 1: Generate Client Certificate

Generate a client certificate for the MCP server:

# Generate CA (if not exists)
openssl genrsa -out ca.key 2048
openssl req -new -x509 -days 365 -key ca.key -out ca.crt \
  -subj "/CN=GP-MCP-CA"

# Generate client private key
openssl genrsa -out client.key 2048

# Generate client certificate signing request
openssl req -new -key client.key -out client.csr \
  -subj "/CN=gp-mcp-server"

# Sign client certificate with CA
openssl x509 -req -days 365 -in client.csr -CA ca.crt -CAkey ca.key \
  -CAcreateserial -out client.crt

Step 2: Configure Authorization Server

Configure your authorization server to accept the client certificate:

Keycloak Example:

  1. Copy ca.crt to Keycloak truststore
  2. Configure Keycloak to require client certificates for JWKS endpoint
  3. Map client certificate CN to client ID (if needed)

Step 3: Configure MCP Server

Add mTLS configuration to your MCP server config

auth:
  oauth2:
    authorization_server_url: "https://keycloak.internal.com/realms/master"
    jwks_url: "https://keycloak.internal.com/realms/master/protocol/openid-connect/certs"
    
    # Enable TLS configuration
    use_tls: true
    
    # CA certificate for verifying authorization server certificate
    tls_ca_file: "/path/to/ca.crt"
    
    # Client certificate for mTLS
    tls_cert_file: "/path/to/client.crt"
    tls_key_file: "/path/to/client.key"

Step 4: Verify mTLS Connection

Test the mTLS connection:

# Test JWKS endpoint with client certificate
curl https://keycloak.internal.com/realms/master/protocol/openid-connect/certs \
  --cacert /path/to/ca.crt \
  --cert /path/to/client.crt \
  --key /path/to/client.key

          - "readonly"
        database_username: "readonly_user"
        database_password: ""
    
    default_database_username: "readonly_user"
    default_database_password: ""

Self-Signed Certificate Configuration

If your authorization server uses a self-signed certificate:

auth:
  oauth2:
    # Enable TLS
    use_tls: true
    
    # CA certificate for verifying self-signed server certificate
    tls_ca_file: "/path/to/keycloak-ca.crt"
    
    # Optional: Client certificate for mTLS
    tls_cert_file: "/path/to/client.crt"
    tls_key_file: "/path/to/client.key"

Troubleshooting mTLS

Issue1: Certificate Verification Failed

  • Error: x509: certificate signed by unknown authority
  • Solution: Ensure tls_ca_file points to the correct CA certificate that signed the authorization server's certificate.

Issue2: Client Certificate Rejected

  • Error: tls: bad certificate or 403 Forbidden
  • Solution:
    • Verify client certificate is signed by a trusted CA
    • Verify authorization server is configured to accept the client certificate
    • Check certificate CN matches expected client identifier (if required)

Issue3: Connection Timeout

  • Error: Connection to authorization server times out
  • Solution:
    • Verify network connectivity to authorization server
    • Check firewall rules allow mTLS connections
    • Verify authorization server is listening on correct port
